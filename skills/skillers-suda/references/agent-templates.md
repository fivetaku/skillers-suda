# 에이전트 템플릿 가이드

skillers-suda가 에이전트 컴포넌트를 생성할 때 참조하는 품질 기준과 템플릿.

## 1. 에이전트 파일 표준 구조

```markdown
---
name: {agent-name}
description: {한 줄 설명 — 에이전트가 트리거되는 상황을 명확히 기술}
model: inherit
tools: [{사용할 도구 목록}]
---

# {Agent Name}

## 역할
{에이전트의 핵심 역할과 목적}

## 전문 분야
{에이전트가 특화된 도메인}

## 작업 방식
{에이전트가 작업을 수행하는 방법}

## 도구 사용 가이드
{각 도구를 언제, 어떻게 사용하는지}

## R&R (역할과 책임)
### 해야 할 것
- ...

### 하지 말 것
- ...
```

## 2. 좋은 에이전트의 특성

| 항목 | 기준 |
|------|------|
| description | 트리거 상황을 구체적으로 기술 (Pushy 전략 적용) |
| 역할 | 명확한 범위 — 하나의 역할에 집중 |
| 도구 | 필요한 도구만 최소한으로 배정 |
| R&R | 해야 할 것과 하지 말 것을 명확히 분리 |
| 분량 | 500-1,500 단어 (스킬보다 짧게) |

## 3. 도구 배정 가이드라인

| 에이전트 유형 | 권장 도구 |
|--------------|-----------|
| 분석/리서치 | Read, Grep, Glob, WebSearch, WebFetch |
| 코드 작성/수정 | Read, Write, Edit, Grep, Glob, Bash |
| 리뷰/검증 | Read, Grep, Glob, Bash |
| 오케스트레이션 | Read, Write, Agent, Bash |

## 4. Description Pushy 전략 (에이전트용)

스킬과 동일한 원칙 적용:
- BAD: `"코드 리뷰 에이전트"`
- GOOD: `"코드 리뷰, 코드 검토, 코드 봐줘, review code — 보안 취약점과 성능 병목을 조기에 발견하기 위해 사용"`

## 5. 안티패턴

- 역할이 모호한 에이전트 ("뭐든지 해주는 에이전트")
- 도구를 과다 배정 (All tools → 필요한 것만)
- description에 역할만 있고 트리거 상황이 없음
- R&R 없이 자유 행동 허용
- model을 불필요하게 opus로 설정 (inherit 권장)
